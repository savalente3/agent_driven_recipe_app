London Workshop · 2025

Building an AI-Driven
Recipe & Meal Planner

A walkthrough of the ideas, decisions, and dead ends behind an app that uses LLMs not just as a feature — but as the product itself.

01 · The Concept

Not a chatbot. Not a search engine.

The starting point was a simple question: what if you could open your fridge, describe what you see, and immediately get a recipe that makes sense for tonight — factoring in the weather, the season, and what you actually feel like eating?

The app had to feel like talking to someone who knows what they're doing, not filling in a form. That distinction shaped every decision that followed.

02 · The Interface

A search bar that thinks

Forms are transactional. A natural-language input is conversational. The decision to use a single free-text field — rather than dropdowns for cuisine, sliders for people, date pickers for range — was about keeping the user in a flow state.

The interface detects intent keywords like Recipe and Meal plan inline, highlighting them as the user types. This gives the feeling that the app is already understanding — before anything is submitted.

The location sits above the search bar rather than inside the form because it's ambient context, not part of the query. It's always there, quietly informing everything.

Recipe

One dish,
right now

Driven by ingredients — ideally from a photo — and the mood of today. The agent acts as a chef who knows the weather and picks what feels right.

Meal Plan

A full week,
thoughtfully

Driven by a date range, preferences, and constraints. The agent acts as a nutritionist who balances variety, health, and what the user actually likes.

04 · The AI Architecture

One agent isn't enough

The first instinct is to give an LLM the user's query and ask for a recipe. That works — barely. The problem is a single agent trying to do too many things at once: understand intent, search the web, factor in weather, write the recipe, and return structured data. The results are generic and inconsistent.

The better approach is a hierarchy. One orchestrator that reads intent and coordinates. Two specialists — a chef and a nutritionist — that do the actual work.

Orchestrator
Reads the query, identifies intent, analyses the image if present, then delegates to the right specialist.
Sub-agent (Chef or Nutritionist)
Receives focused context. Searches for weather, searches for recipes, produces a structured response.
Structured Output
A typed schema the UI can render directly — no parsing, no guessing.
Each agent has one job and does it well.
05 · The Weather Idea

Weather as a creative input, not a utility

Weather in most apps is informational — it tells you what to wear. Here it's a creative signal. A grey rainy afternoon calls for something comforting and indulgent. A bright morning calls for something fresh and light.

The agent doesn't just note the weather — it lets it set the mood of the dish. This is what makes the output feel less like a database result and more like a recommendation from someone who's paying attention.

"The weather outside should change what ends up on the plate."
06 · Meal Plan Structure

Let the agent decide the structure

A meal plan for tomorrow looks different from one for next month. Rather than forcing the user to pick a view (day / week / month), the agent reads the date range and decides the right block structure itself.

1–2 days becomes individual day blocks. A week becomes a single week block. A month becomes a monthly calendar. The UI renders whatever the agent returns — the structure is part of the response, not the request.

1–2 days → Day blocks 3–14 days → Week blocks 15+ days → Month view
07 · What Didn't Work

The dead ends

Every decision came with alternatives that seemed right until they weren't.
Page transitions via the browser View Transitions API
The idea was to animate between states using the native browser API — smooth, hardware-accelerated, no library needed. The problem: Angular updates the DOM asynchronously via signals. By the time the browser captured the "before" snapshot for the transition, Angular had already re-rendered the "after" state. The transition saw no change. Switched to Angular's own animation system which hooks into the rendering cycle directly.
Prefetching weather on page load
To speed up the pipeline, the idea was to fetch the weather as soon as the location is set — before the user even submits. The problem: the date context only exists after the query. "Recipe for tomorrow" needs tomorrow's weather. "Meal plan for next week" needs a week's forecast. You can't fetch the right weather without first reading the intent.
Parallel weather + recipe search within the sub-agent
For the recipe agent, the weather result is needed to craft a good recipe search query — "hearty pasta for a rainy day". Running them in parallel means the search can't be weather-informed. The dependency chain is unavoidable for recipes. For meal planning, it's fine — recipe search there is preference-driven, not weather-driven.
08 · Pipeline Design

Speed through parallelism — where the dependency allows

LLM pipelines are slow by default because each step waits for the previous one. The question is always: which steps actually depend on each other, and which ones are just sequential by habit?

For the meal planner, weather and recipe searches are independent — they can run at the same time. More importantly, while the orchestrator analyses an image for ingredients, it can also search for what's in season locally. These results are then merged and handed to the sub-agent as a single, enriched context — which makes just one search call.

Orchestrator — parallel
Reads image for ingredients · searches weather · searches seasonal produce — all at the same time.
Merge
Ingredients + weather + what's in season → passed to sub-agent as unified context.
Sub-agent — one call
One weather-informed, season-aware recipe search. Then generate.
Map the dependencies first. Parallelise everything else.
09 · UI Follows the AI

The UI is shaped by how LLMs actually behave

Several UI decisions weren't driven by design preference — they were driven by LLM constraints and behaviour.

Structured output means the agent returns typed data the UI renders directly. This dictated the component design — every field in the recipe card, every block in the meal plan, maps to something the agent produces.

Loading states matter more here than in a typical app. An LLM call with tool use takes 5–15 seconds. The UI has to hold the user's attention and communicate that something real is happening — not just spin a spinner.

The tiles on the home screen aren't decorative. They preview the kind of result the user can expect, and they animate into the expanded result when it arrives — creating continuity between the "asking" state and the "answer" state.

10 · What's Next

Where this goes

Seasonal ingredient tool
A new parallel tool at the orchestrator level — what's fresh and in season locally — fed into the recipe search for richer, more grounded results.
Meal plan expanded view
The full calendar UI with requirement bubbles surfacing what the agent understood from the user's query.
Model optimisation
A fast vision model for the orchestrator, a stronger creative model for the sub-agents. Speed and quality aren't always in conflict.
Taking this further inside the company
This app is a prototype for a way of working — agents as product, UI driven by LLM output, pipelines designed around dependencies not assumptions. The same pattern applies far beyond recipes.
